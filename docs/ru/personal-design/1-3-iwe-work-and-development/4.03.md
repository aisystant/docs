#### §4.03 Входной фильтр для нового агента

**Время:** 45 мин чтение + 30 мин

**В одном предложении:** Человек думает, что новый агент подключается просто — скачал, запустил, поговорил, — а на самом деле каждый новый агент без входного фильтра — это потенциальный хаос: он не знает контуров, не понимает ролей, ломает процессы, и пока вы не провели IntegrationGate, вы не подключили агента, а выпустили его в систему вслепую.

**Мем, который снимается.** «Я услышал про новую модель. Она круче текущей. Давайте её подключим — быстро, пока не устарела.» На самом деле новая модель — это не просто «круче». У неё другие ограничения, другой контекст, другие возможности галлюцинаций. Подключение без IntegrationGate — это как нанимать сотрудника без собеседования: может сработать, а может разрушить процессы. IntegrationGate — не бюрократия, а защита инвестиций: вы потратили месяцы на выстраивание системы, и один непроверенный агент может всё разрушить.

**Понятия.** IntegrationGate — пятиступенчатый процесс подключения нового агента. Шаг 1: Тип модели и ограничения — какая модель, какие версии, какие известные проблемы, какие задачи решает хорошо, какие плохо. Шаг 2: Контур работы — какие файлы доступны, какие MCP-серверы подключены, какие операции разрешены, какие запрещены. Шаг 3: Роли — какую роль агент выполняет, какие продукты создаёт, с какими агентами взаимодействует. Шаг 4: Продукты и формат — в каком формате агент создаёт артефакты, какие шаблоны использует, какие критерии приёмки. Шаг 5: Процессы взаимодействия — как пилот обращается к агенту, как часто, какие команды используются, как завершается сессия. Gate предотвращает хаотичное подключение: каждый агент проходит проверку перед тем, как получить доступ к системе.

**Объяснение.** IntegrationGate — не одноразовая процедура. Он имеет два режима: первичный и периодический. Первичный Gate — при подключении нового агента. Периодический Gate — при обновлении модели, изменении MCP-набора, смене роли. Каждое изменение требует перепроверки: новая версия модели может иметь другие ограничения, новый MCP-сервер — открыть новые уязвимости. Gate также включает тестовый период: агент работает под наблюдением, прежде чем получить полный доступ. Тестовый период — не недоверие, а осторожность. Пилот проверяет: агент понимает роль? Соблюдает контур? Создаёт продукты в правильном формате? Только после теста агент получает полный доступ. Без Gate вы рискуете: агент с неправильной ролью начнёт делать чужую работу, агент без контура повредит файлы, агент без формата создаст хаос в документации.

**На практике.** Практика «Описание Gate» (30 мин):

1. Выберите одного потенциального нового агента (модель, которую хотите попробовать). (5 мин)
2. Опишите IntegrationGate: тип модели, контур, роль, продукты, процессы. Используйте шаблон из Pack. (20 мин)
3. Проверьте: Gate покрывает все пять шагов? Есть ли неясности? Зафиксируйте в Pack. (5 мин)

**Типичный кейс.** Команда из трёх человек решила подключить новую модель для анализа данных. Без Gate они просто дали ей доступ к базе данных и сказали: «Анализируй». Модель начала генерировать запросы, которые грузили сервер, и выдавала результаты в формате, несовместимом с текущими отчётами. Через неделю команда потратила два дня на приведение результатов в порядок. После введения Gate: перед подключением определили, что модель хороша для исследовательского анализа, но не для продакшн-отчётов. Ограничили контур: доступ только к копии данных, не к продакшн. Определили роль: Диагност, не Портной. Задали формат: результаты в Markdown с явными оговорками о достоверности. Процесс: агент работает под наблюдением первые три сессии. Результат: интеграция прошла без сбоев, результаты были полезны, а не хаотичны.

**Типичная ошибка.** «Gate — это лишняя бюрократия. Мы agile, мы пробуем быстро.» Agile — не отсутствие структуры, а адаптивная структура. Gate — не запрет на эксперименты, а рамка для экспериментов. Вы можете пробовать быстро, но внутри контура, который защищает систему. Другая ошибка: «У меня один агент, мне не нужен Gate.» Gate нужен не потому, что агентов много. Gate нужен потому, что каждый новый агент — это изменение системы. Даже один новый агент может нарушить баланс.

**Степени мастерства:**

1. Объяснение. Могу назвать пять шагов IntegrationGate и объяснить назначение каждого. Критерий перехода: описал Gate для одного агента.
2. Умение. При подключении нового агента прохожу все пять шагов Gate и фиксирую в Pack. Критерий перехода: три агента подключены через Gate без сбоев.
3. Навык. Системно использую Gate: шаблон Gate в Pack, тестовый период, периодическая перепроверка. Критерий перехода: коллега использовала ваш шаблон Gate для подключения агента.
4. Мастерство. Проектирую Gate для новой среды: определяю шаги, критерии, тесты, критерии приёмки. Критерий перехода: другой человект подключил агента по вашему Gate без вашей помощи.

**Проверка себя.**

- Могу назвать пять шагов IntegrationGate без подсказок
- Описал Gate для каждого агента в моей IWE
- Провожу тестовый период перед полным доступом нового агента
- Проверяю Gate при обновлении модели или MCP-набора
- Могу объяснить новому человеку, как подключить агента через Gate, за десять минут

**На практике.** Если вы собираетесь попробовать новую модель — не начинайте с «Привет, помоги мне». Начните с Gate: определите тип, контур, роль, формат, процесс. Это займёт двадцать минут, но сэкономит часы на исправление последствий.
